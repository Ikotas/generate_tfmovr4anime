<h2><strong>generate_tfmovr4anime.exe</strong></h2>

<p><strong>Description</strong></p>

<p>
    generate_tfmovr4anime is a specialized tool designed to process and clean up TFM (Field Matching) output files, specifically optimized for hand-drawn (cel) animation. Unlike general IVTC tools, this program uses a unique algorithm to maintain stable telecine cycles even during long still-frame sequences or irregular cel-action where pulses are frequently missing.<br>
    By eliminating incorrect match code (c/p) detections and snapping to the earliest consistent boundary, it enables highly accurate and stable field matching for anime sources.
</p>

<p><strong>Target Sources</strong></p>

<ul>
    <li>
        Sources telecined from 23.976 fps to 29.97 fps.
    </li>
    <li>
        Primary Target: Hand-drawn (cel) animation and anime with irregular cycles.<br>
        <i>*Note: For live-action and CG animation, please use </i><a target="_blank" rel="noopener noreferrer" href="https://github.com/Ikotas/generate_tfmovr"><i>generate_tfmovr</i></a><i>.</i>
    </li>
</ul>

<p><strong>Required Files</strong></p>

<p>
    An output file generated by TFM.<br>
    Recommended TFM settings for output:
</p>
<pre>TFM(mode=0,pp=1,slow=2,micmatching=0,output="x:\path\filename.tfm")</pre>
<p>
    <i>*Note: </i><code><i>pp=1</i></code><i> is required.</i>
</p>

<p><strong>Usage</strong></p>

<pre>generate_tfmovr4anime.exe [options] TFM_output_file</pre>

<p><strong>Options</strong></p>

<ul>
    <li>
        <code>-t &lt;int&gt;</code> : <strong>threshold_min_p</strong> (default: 6)
    </li>
    <li>
        <code>-r &lt;float&gt;</code> : <strong>ratio_threshold</strong> (default: 0.9)
    </li>
    <li>
        <code>-c &lt;float&gt;</code> : <strong>consistency_acc</strong> (default: 0.8)
    </li>
    <li>
        <code>-w &lt;int&gt;</code> : <strong>window_size_val</strong> (default: 100)
    </li>
    <li>
        <code>-m &lt;int&gt;</code> : <strong>min_segment_len</strong> (default: 150)
    </li>
    <li>
        <code>-b &lt;int&gt;</code> : <strong>boundary_back</strong> (default: 300)
    </li>
    <li>
        <code>-o &lt;file&gt;</code> : <strong>output filename</strong>
    </li>
</ul>
<hr>

<p><strong>Option Details</strong></p>

<p>
    <strong>-t : threshold_min_p (Minimum Pulse Requirement)</strong>
</p>
<ul>
    <li>
        <strong>Purpose:</strong> Filters out static scenes or noise to ensure the tool only attempts cycle detection when there is enough motion data.
    </li>
    <li>
        <strong>Default:</strong> 6 (frames)
    </li>
    <li>
        <strong>Details:</strong> The tool will only attempt to identify a telecine cycle if the scanning window contains at least this many valid 'p' frames.
    </li>
    <li>
        <strong>Adjustment:</strong> For high-action sources with very few still frames, this can be increased to improve reliability. For slow-paced anime with very limited motion, lowering this value may help detect cycles earlier, though it increases sensitivity to noise.
    </li>
</ul>
<p>
    <strong>-r : ratio_threshold (New Cycle Dominance Threshold)</strong>
</p>
<ul>
    <li>
        <strong>Purpose:</strong> Controls how "convincing" a new cycle candidate must be before the tool considers switching to it.
    </li>
    <li>
        <strong>Default:</strong> 0.9 (90%)
    </li>
    <li>
        <strong>Details:</strong> A new cycle must match the TFM match codes with at least this ratio within the current window to be considered a candidate for transition.
    </li>
    <li>
        <strong>Adjustment:</strong> Lowering this (e.g., to 0.7) makes the tool more aggressive at following rapid cycle changes, but may cause "jittery" transitions in noisy scenes. Keeping it at 0.9 ensures high stability for standard cel animation.
    </li>
</ul>
<p>
    <strong>-c : consistency_acc (Conflict-based Inertia Threshold)</strong>
</p>
<ul>
    <li>
        <strong>Purpose:</strong> Defines the "Inertia" or the strength with which the tool holds onto the current cycle.
    </li>
    <li>
        <strong>Default:</strong> 0.8 (80%)
    </li>
    <li>
        <strong>Details:</strong> This is the core of the anime-optimized algorithm. It calculates accuracy based on Conflicts (where a 'p' appears in a 'c' position) rather than missing pulses. As long as the current cycle's "Conflict-free" rate remains above this threshold, the tool will maintain the cycle even if many 'p' pulses are missing due to still frames.
    </li>
    <li>
        <strong>Adjustment:</strong> Increase to 0.9 or higher if the tool is incorrectly staying in a broken cycle. Decrease to 0.6 if the tool is switching to 'c' or other patterns too easily during scenes with irregular cel-timing.
    </li>
</ul>
<p>
    <strong>-w : window_size_val (Statistical Scanning Window)</strong>
</p>
<ul>
    <li>
        <strong>Purpose:</strong> Sets the frame range used for calculating cycle statistics.
    </li>
    <li>
        <strong>Default:</strong> 100 (frames)
    </li>
    <li>
        <strong>Details:</strong> Determines the breadth of the look-ahead/look-behind used to calculate dominance and consistency.
    </li>
    <li>
        <strong>Adjustment:</strong> A larger window (e.g., 150) provides smoother, more stable results but may delay the detection of a legitimate cycle change. A smaller window (e.g., 50) reacts faster but is more prone to noise-induced errors.
    </li>
</ul>
<p>
    <strong>-m : min_segment_len (Minimum Segment Durability)</strong>
</p>
<ul>
    <li>
        <strong>Purpose:</strong> Prevents frequent, tiny cycle changes (micro-segmentation).
    </li>
    <li>
        <strong>Default:</strong> 150 (frames)
    </li>
    <li>
        <strong>Details:</strong> Ensures that once a cycle is established, it must last for at least this many frames before another switch is permitted. This mimics the reality of anime production where telecine phases typically don't change every few seconds.
    </li>
    <li>
        <strong>Adjustment:</strong> Lower this if the source genuinely has very frequent scene cuts with phase shifts.
    </li>
</ul>
<p>
    <strong>-b : boundary_back (Earliest Fit Search Range)</strong>
</p>
<ul>
    <li>
        <strong>Purpose:</strong> Controls how far the tool "looks back" into the past to find the exact frame where a new cycle started.
    </li>
    <li>
        <strong>Default:</strong> 300 (frames)
    </li>
    <li>
        <strong>Details:</strong> When a cycle change is confirmed, the tool scans backward within this range to find the Earliest Fit-the first frame where the new pattern is consistent with the data. This eliminates "Detection Lag" caused by statistical accumulation.
    </li>
    <li>
        <strong>Adjustment:</strong> If you notice that pattern changes are still happening several frames after a scene cut, increase this value to allow for a deeper backward search.
    </li>
</ul>
<p>
    <strong>-o : Output Filename Specification</strong>
</p>
<ul>
    <li>
        <strong>Purpose:</strong> Customizes the name and path of the generated .tfmovr file.
    </li>
    <li>
        <strong>Details:</strong>
        <ul>
            <li>
                With extension (e.g., -o final.txt): Outputs exactly as named.
            </li>
            <li>
                Without extension (e.g., -o final_list): Automatically appends .tfmovr.
            </li>
            <li>
                Not specified: Defaults to the input filename with the extension replaced by .tfmovr.
            </li>
        </ul>
    </li>
</ul>
<hr>

<p><strong>Core Algorithms for Anime</strong></p>

<ol>
    <li>
        <strong>Conflict-based Inertia</strong><br>
        Standard tools often misidentify still frames as progressive ('c') because they lack the 'p' pulses required for a telecine match. generate_tfmovr4anime treats these "missing pulses" as non-errors. A cycle is only considered "broken" if a "Conflict" occurs (i.e., a 'p' pulse appears where there should be a 'c'). This ensures stability across long pans or still scenes common in cel animation.
    </li>
    <li>
        <strong>Earliest Fit Boundary Snapping</strong><br>
        When a cycle change is detected, the tool scans backward (up to the value set by -b) to find the earliest possible frame where the new cycle fits without contradiction. This eliminates the "Detection Lag" and ensures the new pattern starts exactly at the scene cut.
    </li>
    <li>
        <strong>Wildcard Handling ('h' frames)</strong><br>
        TFM's 'h' frames are treated as wildcards that can be interpreted as either c or p depending on the surrounding cycle requirements, further increasing noise tolerance.
    </li>
</ol>

<p><strong>Encoding Settings</strong></p>

<pre>deinted = nnedi3(field=-1)<br>
TFM(mode=0,pp=3,clip2=deinted,ovr="xxxxx.tfmovr")
</pre>
<p>
    <i>*note: Reasons why a deinterlacer is necessary</i>
</p>
<ul>
    <li>
        <i>Because the first frame is always interlaced when the pattern after a cycle change is pcccp.</i>
    </li>
    <li>
        <i>To clean up any p characters that might have been missed in sections with irregular match codes.</i>
    </li>
</ul>
